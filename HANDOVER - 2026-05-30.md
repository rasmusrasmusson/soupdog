## Platform Vision

Soupdog is not a recipe website. It is a **food execution platform** — a structured knowledge graph of food, processes, and tools that AI can reason over. The long-term goal is software-defined food preparation: deterministic execution, adaptive cooking, food-aware appliances, and household food intelligence.

**The graph is the moat. The AI is the interface.**

Every architectural decision should be made with this in mind. The first public version is intentionally simple, but every table, column, and relation exists to support a platform that can eventually:
- Generate and simulate new recipes and food products before they are cooked
- Produce appliance-specific execution profiles superior to manufacturer instructions
- Optimise commercial kitchen execution across equipment, staff, and timing
- Personalise nutrition at the individual level with cooking-loss-adjusted precision
- Learn variation patterns from curated seed recipes and generalise to the entire food space

### Source + confidence convention
Every content table has `source` and `confidence` columns:
- `source`: `'human_authored'` | `'ai_generated'` | `'ai_generated_reviewed'` | `'imported_usda'` | `'imported_off'`
- `confidence`: `0.0–1.0` (human_authored = 1.0, ai_generated = 0.2–0.5, lab_tested = 0.95+)

This is not optional metadata — it's how AI weights information when reasoning over the graph.

## Key Architectural Principles

**1. Everything has state**  
Ingredients, equipment, and environments have physical states that affect execution. Store temperatures as numbers (°C), not labels. "Frozen" is `-18°C`, not an enum.

**2. Recipes are process graphs, not documents**  
Each step links to a `task` (atomic work unit), has measurable `outcome_criteria`, and knows its required equipment at the generic level. Appliance-specific overrides live in `execution_variants`.

**3. Variants, not versions**  
Changing servings, equipment, method, ingredient, or food state creates an `execution_variant` — not a new `recipe_version`. Versions are for meaningful content changes only. Variants store `divergence_score` (0–1): below ~0.35 = same recipe family, above = significant variant.

**4. The entity graph is the intelligence layer**  
`entity_relations` connects everything. Substitution, scheduling, simulation, and recommendation all query this graph. Populate it carefully — strength and context matter.

**5. Products are first-class citizens**  
A Dr. Oetker frozen pizza cooked in a specific Panasonic oven is a unique data asset. `product_cooking_profiles` is where Soupdog generates data no other platform has. Prioritise this content.

## Recipe Editor Architecture

**File:** `src/components/recipe/RecipeEditor.tsx`

### Structure: Groups → Steps
- Each Group has: `outputName`, `outputQuantityValue` (always grams internally), `outputQuantityUnit` (display), `toolInstances[]`, `steps[]`
- Each Step has: `taskId`, `taskName`, `taskFamily`, `stepIngredients[]`, `stepTools[]`, `instruction`, `durationMinutes`

### Key behaviours
- **Yield** always stored in grams internally; 4 unit buttons (g/kg/ml/l) only change display unit
- **Tasks** loaded from `/api/tasks` — global + personal
- **Custom tasks** saved via `POST /api/tasks`, available cross-session
- **Tool instances** tracked per group with colour dots; smart suggestion via `TOOL_EQUIV_GROUPS`
- **Appliance settings** saved in `version_steps.appliance_settings` JSONB (working)
- **Save bar** fixed to bottom of page

### Data flow on save
```
RecipeEditor → handleSubmit → POST/PUT /api/my/recipes/[id]
  ↓
version_steps (with appliance_settings JSONB)
  ↓ step_id linked
version_ingredients (global ingOrderIndex)
  ↓
legacy recipes mirror (for public /recipes/[slug] display)
```

### Data flow on edit load
```
GET /api/my/recipes/[id]
  → separate queries (not nested joins) to avoid RLS
  → recipe_canonicals → recipe_versions → version_steps → version_ingredients
  → ingredient names fetched separately via .in('id', ingIds)
  ↓
initialToGroups() restores taskId/taskName/taskFamily/stepTools from appliance_settings
```

2. **Live AI scaling still in recipe page** — The servings stepper on `/recipes/[slug]` currently calls the AI scale API on every serving change. This violates the "exact instructions" principle — scaling should produce pre-authored variants, not live AI output. **Fix needed:** Remove the `triggerScale` call from the servings stepper. The stepper should just change the display count. The scale API remains for the future variant authoring flow.

3. **Legacy mirror dependency** — Public recipe pages query the `recipes` table. Newly created recipes auto-mirror on create. Older seed recipes may need manual SQL for mirror rows.

4. **Google OAuth in test mode** — Google Cloud Console project needs publishing for production use.

5. **`/equipment` route naming** — Tools nav item links to `/equipment` but is labelled "Tools". Consider renaming route to `/tools` when building equipment pages.
